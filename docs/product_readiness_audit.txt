====================================
Generated for enterprise hardening. Target: production-grade deployment with NeuronSQL as first-class subsystem.

3. TOOL REGISTRY STRUCTURE
-------------------------------------------------------------------------------

  Base (NewRegistry): sql, http, code, shell, browser, visualization
  With VFS: filesystem
  With hierarchical memory: memory
  With workspace: collaboration
  With NeuronDB client: ml, vector, rag, analytics, hybrid_search, reranking
  Additional: multimodal, web_search
  NeuronSQL module (when enabled): neuronsql.schema_snapshot, neuronsql.validate_sql, neuronsql.explain_json,
    neuronsql.generate_select, neuronsql.optimize_select, neuronsql.plpgsql_generate,
    neuronsql_generate (legacy), neuronsql_optimize (legacy)

  Claw gateway exposes only handlers with prefix "neuronsql." (ListClawTools). Raw "sql" is not exposed to Claw.
  Tool execution: Registry.ExecuteTool -> handler by tool.HandlerType; audit LogToolCall async (bounded concurrency).

-------------------------------------------------------------------------------
4. WORKFLOW ENGINE BOUNDARIES
-------------------------------------------------------------------------------

  Engine: internal/workflow/engine.go. Uses db.Queries, agent.Runtime, tools.Registry, audit Logger, email, webhook, baseURL.
  Step types: agent, tool, approval, http, sql, conditional.
  Execution: Create execution record, load workflow and steps, execute steps in DAG order; approval steps use HITL.
  Idempotency: GetWorkflowStepExecutionByIdempotencyKey for step replay; idempotency key per step.
  Scheduling: workflow_schedules table; cron-based (scheduler in jobs).
  Audit: LogWorkflowRun on execution; approval actions logged via LogApproval.

-------------------------------------------------------------------------------
5. MEMORY SUBSYSTEM LAYOUT
-------------------------------------------------------------------------------

  Hierarchical: working (current session), episodic (recent), semantic (long-term). internal/agent/hierarchical_memory.go, advanced_memory.go.
  Promotion: working -> episodic -> semantic (memory_consolidation.go, worker/memory_promoter.go).
  Vector search: HNSW in NeuronDB for semantic retrieval.
  Cross-session: cross_session_memory.go; share memories across sessions with controls.
  Features: memory_auto_writer, memory_learning, memory_forgetting, memory_corruption, memory_conflict_resolver,
            memory_adaptation, memory_quality, memory_feedback; retrieval_learning, knowledge_router, modular_rag.
  Storage: DB-backed; chunks with embeddings in NeuronDB.

-------------------------------------------------------------------------------
6. SECURITY FLOW DIAGRAM
-------------------------------------------------------------------------------

  Request -> RequestIDMiddleware (X-Request-ID)
          -> RequestTimeoutMiddleware (60s)
          -> AuthMiddleware (Bearer/ApiKey; APIKeyManager, PrincipalManager, RateLimiter; 401/429)
          -> RequestBodyLimitMiddleware (10MB)
          -> SecurityHeadersMiddleware
          -> CORSMiddleware(cfg)
          -> LoggingMiddleware
          -> Handler (role checks via auth.RequireAnyRole where needed; admin for /admin/*)
  Tool execution: context has agent_id, session_id, principal; AuditLogger.LogToolCall after execute.
  NeuronSQL: PolicyEngine (blocklist, classifier, sanitizer) before any SQL run; SafeConnection read-only, timeouts.

-------------------------------------------------------------------------------
7. OBSERVABILITY INTEGRATION POINTS
-------------------------------------------------------------------------------

  Prometheus: internal/metrics/prometheus.go. Counters/histograms for HTTP (method, endpoint, status), agent (agent_id, status),
              LLM (model, status; tokens by model, type), memory (chunks stored, etc.), tool calls, workflow (runs, duration).
  Logging: metrics.InfoWithContext, DebugWithContext, WarnWithContext, ErrorWithContext; log_context.go WithLogContext (request_id, etc.).
  Tracing: observability/tracing.go (stubs present); metrics/tracing.go.
  Diagnostics: GET /api/v1/admin/diagnostics returns version, DB health, module health (AdminHandlers).
  Request ID: Set in middleware, passed in context, included in error responses and response header.

8. GAPS FOR ENTERPRISE DEPLOYMENT (RANKED)
-------------------------------------------------------------------------------

  P0 (Required for enterprise)
    - Explicit API version response header (X-API-Version: v1).
    - OpenAPI spec complete for all routes (NeuronSQL, Claw, admin, memory management, batch, workflows/schedules).
    - Multi-tenancy: org_id, organizations table, org-level isolation and quotas.
    - RBAC expansion: tool-level, workflow-level, workspace-level permissions; approval roles.
    - Audit upgrade: dedicated tables (audit_events, policy_decisions, tool_invocations, workflow_executions, approval_actions);
      org_id, workspace_id; retention and partitioning.
    - Compliance profiles: standard, gdpr, hipaa, sox; memory TTL, audit verbosity, export restrictions.
    - NeuronSQL: strict SQL length/charset validation; all tool outputs include citations, validation_report, policy_decision.

  P1 (Required for SaaS)
    - Request/response schema validation (RejectUnknownFields wired).
    - Idempotency keys for tool calls and background jobs; workflow step time budgets.
    - Circuit breaker per tool; tool timeouts; workflow max duration; dead-letter queue.
    - Graceful shutdown: drain in-flight, flush audit, persist snapshots.
    - Structured logging standard: request_id, org_id, workspace_id, agent_id, tool_name, workflow_id, latency_ms, status, cost.
    - Prometheus: policy_blocks_total, memory_promotions_total, queue_depth, cost_per_workspace; tool latency by tool_name.
    - Distributed tracing spans for LLM, tool chain, workflow steps, memory retrieval, NeuronSQL pipeline.
    - Deterministic execution mode (tool-only, seed-controlled, replayable).

  P2 (Scaling / optimization)
    - Per-workspace rate limit and burst.
    - HPA and resource limits in Helm; values.schema.json; existingSecret for DB_PASSWORD; PVC.
    - Terraform module for Postgres, service, load balancer, monitoring.
    - Production Dockerfiles: multi-stage, non-root; CUDA/ROCm/Metal variants.

  P3 (Ecosystem)
    - Agent blueprints (built-in templates); governance dashboard APIs (costs, tool risk, policy blocks, memory growth, performance).
    - Marketplace: signed packages, version pinning, security rating.
    - CI: openapi-validate, security-lint, deps-scan, integration test harness; fail on undocumented endpoints, missing audit.
